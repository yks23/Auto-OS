# StarryOS 自举（Self-Hosting）完整说明

本文面向当前 `yks23/Auto-OS` 主线进展，给出一份可直接用于团队同步与后续执行的完整文档，覆盖：

1. 路线规划  
2. 遇到的问题与主要完成部件  
3. 接下来改进方案

---

## 1) 路线规划

## 1.1 总目标

在 guest 内完成从源码到内核产物的闭环构建，并验证 guest 产物可独立启动，最终达到：

- **S3（自举）**：guest 内编译 StarryOS kernel ELF，并能在 host QEMU 启动成功。
- **S4（可复现）**：guest build 与 host build 产物达到可比对的一致性目标（sha256 对齐为理想终态）。

## 1.2 分阶段路线（M0 → M6）

| 阶段 | Milestone | 验收核心 |
|---|---|---|
| Phase 0 | M0 | host 可稳定编译内核 ELF（多架构） |
| Phase 1 | M1 | T1-T5 集成后，host 侧 kernel + selfhost tests 可编译 |
| Phase 1.5 | M1.5 | guest 内运行 acceptance tests，达到可用通过率 |
| Phase 2 | M2 (S0) | guest 内可编译并运行 `hello.c` |
| Phase 3 | M3 (S1) | guest 内可编译较大 C 项目（BusyBox） |
| Phase 4 | M4 | guest 内可编译 musl libc |
| Phase 5 | M5 (S2) | guest 内 `cargo build` 跑通（Rust 工具链闭环） |
| Phase 6 | M6 (S3/S4) | guest 编译内核 + 产物启动验证 + 可复现推进 |

## 1.3 当前主线位置（基于现有仓内文档）

- M0 / M1 已完成（host 侧能力稳定）。
- M1.5 已完成并达到 31/31 PASS 的阶段性结果。
- M2 已完成（guest 编译并运行 C 程序）。
- M5 已完成（guest 内 `rustc/cargo` 全链路可用，cargo 构建成功）。
- M3-real / M4 / M6 仍在推进中（从“可运行”迈向“完全自举 + 可复现”）。

---

## 2) 遇到的问题和主要完成的部件

## 2.1 关键问题（阻塞路径上的真实瓶颈）

1. **进程创建链路问题（fork/execve/wait4）**  
   早期导致 shell 外部命令挂死，`make/gcc/cargo` 无法形成可用流水线。

2. **console RX 输入链路问题**  
   guest 无法稳定接收 host 注入输入，交互调试效率极低。

3. **“假成功” syscall 风险**  
   部分接口返回 `Ok(0)` 但未真实实现，导致构建系统/工具链静默失败或数据一致性风险。

4. **vfork / posix_spawn / aspace 切换细节问题**  
   在 cargo → rustc → cc → ld 多级子进程链中暴露，属于 self-hosting 的深水区问题。

5. **跨架构稳定性与资源约束问题**  
   包括栈、内存上限、rootfs 体积、工具链依赖等系统性问题。

## 2.2 主要完成的部件（主线价值最大）

### A. Phase 1 关键能力（T1-T5）

- 多线程 `execve` 支撑
- 文件锁（`flock` + `fcntl` record locks）补齐
- IPv6 v4-mapped 支撑
- `ext4` 挂载能力补齐
- 资源限制与 `prlimit64` 强化

这些能力构成了“工具链可运行”的基础平台。

### B. Selfhost rootfs 与工具链路径建设

- 建立 selfhost rootfs 构建路径（含 C 工具链与后续 Rust 工具链扩展）。
- 打通 guest 内编译与运行的最小闭环（M2）。

### C. Rust 路径打通（M5）

- 在 guest 内完成 `rustc` 与 `cargo build` 实证。
- 跑通 `cargo → rustc → cc → ld → run` 全链路。
- 证明内核在进程、内存映射、加载与 I/O 方面已具备更高实用强度。

### D. 测试与验收工程化

- 建立 selfhost 测试集合与统一输出规范（PASS/FAIL）。
- 按 Milestone 定义可执行验收口径，避免“只看合并不看实跑”。

## 3) 接下来改进的方案

## 3.1 目标导向原则

1. **先闭环，再扩展**：优先完成 M6.S3 自举闭环，再追求 S4 bit-level 可复现。  
2. **先真实语义，再接口覆盖**：减少 dummy/no-op/假成功行为。  
3. **先可回归，再提性能**：先把 CI 与验收链路固化，再做优化。  

## 3.2 近期（短周期）改进项

1. **完成 M3-real（BusyBox 全量构建）**  
   把当前“可编部分 C 项目”提升为“可编完整大体量 C 工程”。

2. **推进 M4（musl libc 构建）**  
   用工具链核心工程验证 ABI 与系统调用细节稳定性。

3. **消化已知遗留失败项**  
   对当前已知 FAIL 测试建立逐项闭环（复现、修复、回归、归档）。

4. **补齐阻塞 syscall 与 procfs/prctl 等高频依赖点**  
   优先处理直接影响 gcc/rustc/cargo/调试链路的能力缺口。

## 3.3 中期（自举阶段）改进项

1. **达成 M6.S3：guest 编译内核并启动验证**  
   明确以“guest 产物可启动”作为唯一硬验收条件。

2. **推进 M6.S4：可复现工程化**  
   固定构建输入、时间源、环境变量与工具链版本，建立 sha 对比流程。

3. **双架构主线并行（riscv64 + x86_64）**  
   避免单架构成功掩盖架构特定缺陷。

## 3.4 工程治理改进（保障长期迭代）

1. **CI 分层固化**  
   将 M1.5 / M2 / M5 / M6 的关键验证动作纳入常规工作流，形成周常回归。

2. **问题分级与 owner 机制**  
   继续按“Foundation / Tier-1..5”管理问题，确保每项有 owner、验收标准、回归用例。

3. **文档与状态统一出口**  
   把“路线、状态、风险、下一步”集中维护，减少多文档之间的信息漂移。

---

## 结论

主线已从“能启动”跨过“能在 guest 内编译并运行 C/Rust 程序”的关键门槛，进入“向完整自举（M6）收口”的阶段。  
后续重点不是再做零散功能扩张，而是围绕 **M3-real → M4 → M6.S3/S4** 做闭环推进，把能力从“可演示”固化为“可持续、可回归、可复现”。
