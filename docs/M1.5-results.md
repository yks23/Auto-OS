# M1.5 验收结果（Phase 1 真验证）— 第二轮（绕过 stdin 方案）

**目标**：在 guest QEMU 内跑 Phase 1 全集（T1-T5）整出来的 31 个 user-space 测试，确认 ≥ 25 个 PASS。

**第二轮实测结论**：**FAIL — 不达标，但根因更聚焦了**：starry kernel 的 **fork+exec 全部死锁**（不只是 stdin，是用户态进程创建本身的问题）。31 个测试 0 PASS，但**这次明确了根因不在 T1-T5 patches，在 starry 上游既有的 process creation 路径**。

## 测试方法变更（绕过 stdin）

第一轮发现 starry kernel console RX 不工作（host 串口字节进不去 BusyBox stdin），所以本轮改用「不依赖 stdin」方案：

- **新增 `patches/M1.5/0001-feat-starry-init-auto-run-opt-run-tests.sh-hook-for-.patch`**：改 starry `starryos/src/init.sh`，在 boot 进 shell 前先检查 `/opt/run-tests.sh`，存在就 exec
- run-tests.sh 用 `cmd & pid=$!; sleep loop; kill -9` 实现"软 timeout"（starry 没有 `timeout(1)`）
- 输出到串口由 host 被动收集

## 关键诊断结果

### 用最简化 init.sh 验证 starry 用户态能力

```sh
echo "===DEBUG-A: builtin echo work==="    # ✅ 工作
echo "===DEBUG-B: builtin pwd==="
pwd                                          # ✅ 工作（builtin）
echo "===DEBUG-C: try external ls==="
ls / 2>&1                                    # ❌ 死锁
echo "===DEBUG-D: ls done==="                # 永远到不了这里
echo "===SELFHOST-DONE==="
```

实际看到的 guest 串口输出（每行 \r\n 已正常）：
```
Welcome to Starry OS!
===DEBUG-A: builtin echo work===
===DEBUG-B: builtin pwd===
/root
===DEBUG-C: try external ls===
[死锁 — 永久 stuck，host 等 60s 也不见后续]
```

### 根因

**starry kernel 在 `/bin/sh` 里 fork + exec 子进程时整体死锁**：
- builtin command（echo / pwd / cd / export）：✅ 工作
- 任何 external command（ls / cat / 测试二进制）：❌ 死锁

具体可能：
1. **fork 不返回**：starry 的 fork 实现不完整，子进程创建时父进程 wait 死锁
2. **execve 不返回**：fork OK 但 child 在 execve 时挂掉，父 waitpid 永等
3. **wait4/waitpid 死锁**：fork+execve 都 OK，但 waitpid syscall 在 starry 实现里有 bug

需要进一步用 strace 类工具诊断，但 starry 没有 ptrace。

### 与 T1-T5 patches 的关系

- T1 修的是**多线程 process** 调 `execve` 的死锁。但**单线程 fork+exec 路径不在 T1 范围内**——它走的是 sh fork → child execve（每个 child 是单线程的）。
- 这个死锁**不是 T1-T5 引入的**：去掉 T1-T5（用 PIN c7e88fb3 原 starry）大概率也死锁。是 starry 原始 process creation 路径上的某个问题（可能 ptrace stop 标志位、可能 child run queue、可能 vfork wait）。

## Followup 优先级（M1.5 重做的前置）

### Followup-α：诊断 starry fork+exec 死锁（D1，最高优先）

任务包要点：
1. 写最小 C 程序：`fork()` → child `_exit(42)` → parent `wait4(child, &ws)`，直接 cargo run 看是否死锁（不通过 sh）
2. 如果死锁：starry kernel 的 `wait4` 或 task 退出 reaping 路径有 bug
3. 如果不死锁：那么是 `execve(/bin/ls)` 在 child 里死掉，看 ELF loader / page fault 路径
4. 用 starry 自己的 `LOG=debug make run` 看 boot 日志

### Followup-β：并行做 stdin/console RX 修复（D1）

第一轮发现的 console RX 问题独立存在，最终也要修。fix 后用户能用 BusyBox 真交互。

- 如果只修 fork+exec：测试可以跑出来，能看到每个 syscall fix 的真 PASS/FAIL，开始判断 T1-T5 各 patch 的实际正确性
