### (3) ⚠️ TTY 阻塞 = 真 starry kernel bug

QEMU 串口 TCP socket 连进去后：
- Python `recv()` 拿到 `Welcome to Starry OS!` + prompt：✅
- Python `sendall(b"sh /opt/run-tests.sh\r\n")`：✅
- guest 端没收到任何字节 → BusyBox shell 永远在等输入，最终 TIMEOUT

我（Director）手工诊断也是同样症状：用 nc / socket 单独连 4444，发任何字节都没回应。

可能根因（按嫌疑度）：
1. **starry kernel 的 console driver TX 与 RX 不对称**：能写 stdout，但 RX path 没被正确路由到 BusyBox 的 stdin（`/dev/console` 或 `/dev/ttyS0`）
2. **BusyBox shell 的 stdin O_NONBLOCK + select/poll 配合 starry kernel 的 ppoll 实现不一致**
3. **T5 patch 的 RLIMIT_NOFILE → AX_FILE_LIMIT=1024 改动**让 BusyBox 的 stdin fd 行为变化（小概率）
4. UART RX 中断没注册或 IRQ 路由错（BusyBox 用 blocking read 而不是 poll，那 RX 中断不就发不到 process？）

### (4) Subagent 自我诊断了 5 个真问题（成果保留在 scripts/）

D4 (Test & CI) subagent **没失败**，它实际推进了非常多基础设施：

| 问题 | subagent fix |
|---|---|

| `phys-memory-size=128M` vs `-m 4G` 不一致 → S-mode page fault | 改 `-m 128M -smp 1`（注释里写明） |

## 失败用例分析

**所有用例同根**：starry kernel 的 BusyBox shell 在串口 stdin 不响应。
- 测试在断言什么：N/A，根本没机会跑
- guest 实际表现：boot OK，进 prompt OK，prompt 后 send 任何字符都不回应
- **推测根因**：starry kernel UART RX 中断或 console line discipline 与 ax-net-ng / 当前 rootfs 的 BusyBox 不兼容

## 怎么修（给后续 subagent 的合同）

### Followup-1：诊断 starry tty stdin 阻塞（D1 责任）

任务包要点：
1. 用最简的 starry 用户态程序（不依赖 BusyBox）测试：fork 一个 child 跑 `read(0, buf, 1)` → 看是否收到 host 通过串口送进来的字节

4. 必要时改 starry kernel 让 stdin 走 polling fallback
