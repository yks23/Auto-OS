09

多核暴露的内核实现问题

核心不是脚本，而是调度和迁移语义

用户态抢占

  • uctx.run() 因 timer 返回后原来没有显式调度点
  • 长 CPU 段可能让 guest heartbeat 长时间不推进
  • 修正：ReturnReason::Interrupt 中 yield_now()

用户任务迁移

  • 直接跨 CPU wake 曾触发 current().as_thread() panic
  • 原因：StarryOS 用户线程上下文尚未完全 migration-safe
  • 修正：用户态运行 pin CPU，blocked wake 回原 CPU

run queue 负载

  • 新增每 CPU run queue load 统计
  • kernel task 可按负载分配
  • 用户任务先保持亲和性，correctness-first
讲法：这一页只讲 OS 实现：用户态抢占点、任务迁移亲和性、run queue 负载。脚本只是把这些验证出来，不是 PR 主体。
10

必须讲清楚的风险

QEMU RISC-V MTTCG 不能当 correctness 证明

速度实验

-smp 4 -accel tcg,thread=multi

可以观察真实并行是否能加速。

正确性优先

-smp 4 -accel tcg,thread=single

避免 RISC-V MTTCG LR/SC reservation 风险。

内核 context-switch 里的 LR/SC 清理 fix 仍然有意义，但它不能修复 QEMU MTTCG 的跨 hart reservation 建模问题。

讲法：这里要主动说明风险。用 thread=multi 跑快了，只能说明速度潜力，最终正确性需要另一个验证路径。

最新状态：v20 的 jobs=2 subset 已 PASS；v21 full early 到 syn 后 heartbeat 消失；v22 已让 starry-kernel 早期 lib 阶段启用 smp feature，并暴露 StoreFault。

讲法：现在不把 jobs=4 当最终结论，也不声称完整多核 M6 已 PASS。重点是多核已经从“能启动”推进到“能复现实打实的 OS 调度压力”。

内核改进与当前证据

每个改动都对应一个 SMP 现象

内核点 遇到的问题 解决策略 当前证据
用户态抢占 CPU-bound rustc 段中 guest 反馈可能消失。 timer interrupt 返回用户循环时显式 yield_now()。 v19 guest heartbeat 持续输出，已越过 v18 的 thiserror 卡点；v21 暴露 jobs=2 heartbeat stall。
任务迁移 blocked 用户任务跨 CPU wake 后 syscall 当前线程上下文错误。 用户态运行 pin CPU，带 TaskExt 的任务唤醒回原 CPU。 v17 的 current().as_thread() panic 不再复现。
run queue SMP 下缺少可解释的 CPU 负载选择信号。 维护 run queue load；非 pinned kernel task 可按负载分配。 为后续 jobs=2/4 压力测试提供基础。
诊断 长跑失败只看最后 panic，定位太慢。 track_caller 记录 preempt/block 调用点。 后续调度 panic 能直接回到内核调用点。
feature wiring 早期 starry-kernel lib 阶段没有覆盖 SMP 调度代码。 新增 smp = ["ax-feat/smp"]。 v22 已打印 --features smp；随后暴露 StoreFault。
讲法：这一页回答“OS 实现改进了什么”。不要把 Docker、QEMU、rootfs 讲成主线，它们只是验证这些内核改动的环境。
